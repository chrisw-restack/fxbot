# Python FX Trading Bot

## Overview

This project implements a **modular Python trading bot** for Forex (FX) markets using **MetaTrader 5 (MT5)** for execution. It is designed for both **live trading** and **backtesting**, with scalability for **multiple strategies, symbols, and risk configurations**.

Key design principles:

- **Separation of responsibilities**: strategy, risk, execution, data, portfolio, and backtest layers are fully decoupled.
- **Event-driven architecture**: strategies and risk modules respond to internal events generated from market data.
- **Reproducible backtesting**: historical data is fetched once from MT5, stored locally as CSV, and used repeatedly in backtests.
- **Synchronous execution**: the event loop is synchronous (not asyncio). Per-bar polling is sufficient.

## Layer Responsibilities

| Layer | Responsibility |
|-------|----------------|
| **Data Layer** | Fetch live or historical OHLC data per symbol/timeframe. Store locally as CSV for reproducible backtests. |
| **Strategy Layer** | Generate `BUY` or `SELL` signals based on market data. Each strategy declares which timeframe(s) and order type it uses. Fully isolated from execution, risk, and data layers. |
| **Risk Layer** | Receives a signal and calculates lot size (dynamic or fixed) and take-profit price (SL distance ├Ч R:R ratio). Stop-loss price is always provided by the strategy. |
| **Execution Layer** | Places orders via MT5 (live) or simulates fills (backtest). Both implement the same base interface. |
| **Portfolio Layer** | Tracks open positions. Enforces conflict blocking, max open trades, and max daily loss. |
| **Backtesting Layer** | Replays historical CSV data bar-by-bar through the full pipeline. Produces trade log and performance summary. |
| **Event Queue / Engine** | Synchronous queue distributing `NEW_BAR` events. One event fired per symbol/timeframe when a new bar closes. |


## Event & Signal Schemas

### NEW_BAR Event
Generated by the data layer each time a new bar closes on a subscribed symbol/timeframe:
```python
@dataclass
class BarEvent:
    symbol: str        # e.g. 'EURUSD'
    timeframe: str     # e.g. 'H1'
    timestamp: datetime
    open: float
    high: float
    low: float
    close: float
    volume: float
```

### Signal
Returned by a strategy's `generate_signal(event)` method. Returns `None` for no trade:
```python
@dataclass
class Signal:
    symbol: str
    direction: str      # 'BUY' or 'SELL'
    order_type: str     # 'MARKET' or 'PENDING'
    entry_price: float  # Current price for MARKET; specific level for PENDING
    stop_loss: float    # Price level тАФ always set by the strategy
    strategy_name: str
    timestamp: datetime
```

- **Stop-loss** is always a price level, calculated by the strategy (e.g. recent swing high/low, or a fixed pip offset from entry).
- **Take-profit** is calculated by the risk manager from `SL distance ├Ч R:R ratio`.
- **Lot size** is calculated by the risk manager based on the chosen sizing mode.
- There is no `CLOSE` signal. Trades always run to their stop-loss or take-profit.


## Risk Management

- **Stop-loss** is always set by the strategy as a price level.
- **Take-profit** is derived by the risk manager: `entry ┬▒ (SL distance ├Ч R:R ratio)`.
- **Default R:R ratio**: 1:2 (configurable per strategy and in backtests).

### Lot Sizing Modes (configured in `config.py`)

| Mode | Behaviour |
|------|-----------|
| `DYNAMIC` | Lot size calculated from: `(account balance ├Ч risk %) ├╖ (SL distance in pips ├Ч pip value)`. Default risk: 0.5% per trade. |
| `FIXED` | A fixed lot size is always used (e.g. `0.01`), regardless of SL size or account balance. |

One mode is active at a time тАФ they are never mixed.


## Portfolio & Conflict Management

Instruments: **EURUSD, GBPUSD, AUDUSD, NZDUSD, USDJPY, USDCAD**

- **One position per symbol at a time**: if a symbol already has an open position, any new signal for that symbol (same or opposite direction) is blocked.
- **Max open trades**: default 4. New signals are blocked when limit is reached.
- **Max daily loss**: default 2% of account balance. No new trades placed for the rest of the day if hit.


## Multi-Timeframe Support

Each strategy declares which timeframe(s) it requires. The event engine subscribes to all needed symbol/timeframe combinations.

Supported timeframes: **M5, M15, H1, H4, D1**

A multi-timeframe strategy (e.g. H4 for trend, H1 for entry trigger) receives events on both timeframes and manages its own internal state to combine the signals.


## Order Types (per strategy)

Each strategy uses either **market orders** or **pending orders** тАФ not both within the same strategy. The strategy declares this via the `order_type` field on the signal.

| Order Type | Behaviour |
|------------|-----------|
| `MARKET` | Entry at current bid/ask at signal time. `entry_price` is the current price. |
| `PENDING` | Entry at a specific price level defined by the strategy. The execution layer infers the pending order type (Buy Stop, Buy Limit, Sell Stop, Sell Limit) from the direction and entry price relative to current price. |


## Backtesting Data Flow

1. Fetch historical OHLC data from MT5 for each symbol/timeframe.
2. Save locally as CSV: `data/historical/<SYMBOL>_<TF>_<YYYYMMDD>-<YYYYMMDD>.csv`
   - Columns: `time, open, high, low, close, tick_volume`
3. Backtest engine reads CSV files bar-by-bar. MT5 is not queried during a backtest.
4. Each bar fires a `NEW_BAR` event through the full pipeline (strategy тЖТ risk тЖТ simulated execution тЖТ logger).
5. Simulated fills assumed at the **open of the next bar** (no look-ahead bias).
6. Trade log and performance summary produced at the end.


## Backtest Output

### Trade Log (one row per trade)

| Column | Description |
|--------|-------------|
| `datetime` | When the trade was placed |
| `symbol` | Instrument |
| `direction` | BUY or SELL |
| `result` | WIN or LOSS |
| `r_multiple` | Outcome in R (e.g. `+2.0` for a 2R win, `-1.0` for full stop-loss) |
| `strategy` | Strategy name |

### Performance Summary

- Total trades taken
- Win rate %
- Total R gained/lost
- Profit factor (gross profit ├╖ gross loss in R)
- Max drawdown (in R and %)
- Expectancy (average R per trade)
- Best win streak / worst loss streak
- Average R on winning trades / average R on losing trades

## Adding New Strategies

1. Create `strategies/<name>.py`.
2. Implement a class with:
   - `TIMEFRAMES: list[str]` тАФ timeframes this strategy subscribes to (e.g. `['H1', 'H4']`)
   - `ORDER_TYPE: str` тАФ `'MARKET'` or `'PENDING'`
   - `generate_signal(event: BarEvent) -> Signal | None`
3. Stop-loss must always be set as a price level on the returned `Signal`.
4. No broker calls, risk calculations, or portfolio logic inside strategy code.
5. Register the strategy in the engine's subscription list in `main_live.py` / `backtest_engine.py`.


## Platform Note

The `MetaTrader5` Python package runs on **Windows only**. Live trading requires a Windows environment with MT5 installed (e.g. a Windows VPS). Backtesting using pre-downloaded CSV files can run cross-platform.
